# 题材趋势联动与桌面自动更新设计

## 目标

第1项“一键小说正文生成”的题材推荐必须先选择题材类型，再按该类型生成 2026 年近期趋势题材，并只提供匹配该类型的文风。小兔助手 Windows 安装版必须在后续代码同步后能够自动获得新版本，不要求朋友反复手动索取安装包。

## 题材推荐交互

推荐区保留现有卡片布局，在“推荐文风”前新增“题材类型”下拉框。默认类型为“都市男频”，支持以下分类：

- 都市男频
- 情感女频
- 古风权谋
- 古风情感
- 年代家庭
- 现实烟火
- 职场商战
- 系统脑洞
- 玄幻仙侠
- 悬疑推理
- 末世科幻
- 萌宝治愈
- 非遗文旅
- 轻喜剧

切换题材类型后立即完成三项联动：

1. 推荐文风下拉框只保留该类型适用的文风。
2. 当前文风自动切换为该类型的首选文风，不沿用不匹配的旧选项。
3. 本地兜底推荐立即切换为该类型的题材池，避免刷新失败时显示其他赛道。

点击“刷新推荐”时，在线 AI 只能返回当前所选类型。每条结果增加 `style` 字段，且必须来自当前类型允许的文风列表。“一键填入大纲”同时写入故事大纲和该条推荐文风。

## 2026 近期趋势规则

在线刷新提示词必须包含运行时当前日期、当前题材类型、最近 30 至 90 天窗口、允许文风列表和平台安全约束。结果要求 4 至 6 条，每条包含：

- `title`：具体题材名，避免空泛赛道名。
- `summary`：解释近期受众情绪和核心吸引力。
- `outline`：可直接扩写的原创大纲，不能照搬具体作品。
- `tags`：3 至 5 个题材标签。
- `style`：当前类型允许的一个文风。

“实时”在本功能中的准确含义是：用户点击时，使用当前配置的在线模型、当前日期和近期趋势窗口即时生成。由于抖音和红果没有接入官方公开榜单 API，界面不得声称结果是官方榜单排名，也不得伪造作品名、播放量或平台数据。在线调用成功时显示生成时间与“联网 AI 近期趋势推荐”；失败时明确显示“本地 2026 趋势兜底”，不把本地数据冒充实时结果。

本地兜底题材池按上述 14 类分别维护，每类至少 4 条，内容面向 2026 年仍可持续创作的主流方向。兜底卡片同样带匹配文风。

## 数据边界

新增独立题材趋势模块，负责：

- 题材类型定义。
- 类型到文风的映射。
- 本地兜底题材池。
- AI JSON 结果校验与文风归一化。
- 根据当前类型筛选推荐。

`Workspace` 只负责保存当前选择、发起在线刷新和渲染结果，不继续堆积题材业务常量。当前题材类型保存在第1项步骤输入中，项目保存后能够恢复。

## 桌面自动更新

采用公开仓库 `xiaotushuoman-ship-it/novel-script-tool` 的 GitHub Releases 作为更新源，使用 `electron-updater`：

1. 安装版启动后延迟检查更新，避免影响首屏打开。
2. 有新版本时后台下载；下载完成后弹出中文确认框。
3. 用户选择“立即更新”后退出、安装并重启；选择“稍后”则本次继续使用旧版。
4. 检查或下载失败不阻塞软件使用，只记录日志并在下次启动重试。
5. 开发模式不检查更新，避免本地开发误触发安装器。

安装包继续使用 NSIS x64。GitHub Release 必须包含安装器、blockmap 和 `latest.yml`，供差分更新和版本判断使用。为降低下载地址编码问题，Release 产物文件名使用 ASCII，例如 `Xiaotu-Assistant-Setup-2026.7.12.exe`；软件名称和快捷方式仍显示“小兔助手”。

## 同步发布流程

新增 GitHub Actions Windows 发布工作流。每次代码同步到 `main` 后自动执行：

1. 安装依赖并运行测试。
2. 生成生产网页构建。
3. 在 CI 中生成单调递增的版本号 `年.月.GitHub运行序号`，例如 `2026.7.12`。
4. 构建 Windows x64 NSIS 安装包。
5. 创建对应 GitHub Release，并上传安装器、blockmap 和 `latest.yml`。

同一次推送仍由 Vercel 自动更新网页端。因此以后用户发出“同步更新”指令时，固定含义为：完成测试后推送 `main`；Vercel 更新网页端，GitHub Actions 更新安装版。普通本地修改不自动推送，也不自动发布，继续遵守用户现有习惯。

首次启用自动更新时，需要发布一个高于当前手工安装包 `0.1.0` 的 Release。由于旧的 `0.1.0` 安装包没有内置更新器，使用者必须手动安装这次的新版本一次；从新版本开始，后续 Release 才能在启动时自动识别并进入更新流程。

## 错误处理

- AI 返回非 JSON、类型不匹配、文风不在允许列表或字段缺失时，丢弃无效条目；全部无效则使用当前类型本地兜底。
- 在线刷新失败不能清空当前卡片，先显示当前类型兜底，再更新状态提示。
- GitHub 无网络、Release 不存在或更新元数据异常时，软件正常启动并保留旧版本。
- 下载中途失败时不执行安装，避免破坏现有版本。
- 发布工作流测试失败时不得创建 Release。

## 测试要求

- 单元测试覆盖题材类型到文风映射、默认文风、AI 结果过滤和本地兜底筛选。
- 组件测试覆盖切换类型后文风选项变化、默认文风联动、刷新提示词包含类型与当前日期、一键填入同时写入大纲和文风。
- Electron 测试覆盖开发模式跳过更新、生产模式调用更新检查、下载完成后的重启安装选择。
- 发布配置测试覆盖 GitHub provider、ASCII 安装包名、`latest.yml` 所需配置和工作流触发条件。
- 最终验证包含完整测试、网页构建、Windows 安装包构建以及安装后启动检查。

## 不在本次范围

- 不抓取或破解抖音、红果的非公开接口。
- 不展示未经官方数据源验证的播放量、热度排名或具体爆款作品榜单。
- 不改变第1项正文生成的总章数、单章字数、实时输出和正文规则。
- 不改变生图备用通道、字字动画和其他步骤的业务逻辑。
